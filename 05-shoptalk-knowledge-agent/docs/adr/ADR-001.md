# ADR-001: FAISS over ChromaDB for Vector Index

**Project:** P5: ShopTalk Knowledge Agent
**Category:** Tool Choice
**Status:** Accepted
**Date:** 2026-03-03

---

## Context

P5 is an experiment platform comparing 35+ RAG configurations: 5 chunking strategies × 3 embedding models × 3 retriever types × hyperparameter sweeps. Each configuration needs its own isolated index — different embedding dimensions (MiniLM=384, MPNet=768, OpenAI=1536), different chunk sets, explicit persistence for reproducibility.

P4 (Resume Coach) used ChromaDB for its REST API. That experience revealed two problems:
1. ChromaDB uses a module-level `EmbeddingFunction` singleton that conflicts when multiple collection configurations coexist in the same process.
2. ChromaDB abstracts embedding injection behind `query_texts`/`query_embeddings` with subtle normalization behavior (calls `normalize_embeddings()` internally, returning `list[np.ndarray]` not `list[list[float]]`).

For an experiment platform, explicit control over the index lifecycle is essential. We need to: create an index with a specific dimension, add vectors, swap the index out, compare retrieval quality, and persist each configuration separately.

---

## Decision

Use **FAISS** (`faiss-cpu`) with `IndexFlatIP` (inner product on L2-normalised vectors = cosine similarity).

Each experiment config creates its own FAISS index:
```python
import faiss
import numpy as np

# Create index for embedding dimension 384 (MiniLM)
index = faiss.IndexFlatIP(384)

# Add normalised vectors
embeddings_norm = embeddings / np.linalg.norm(embeddings, axis=1, keepdims=True)
index.add(embeddings_norm.astype(np.float32))

# Search: returns (distances, indices) — distances = cosine similarity
scores, indices = index.search(query_norm.reshape(1, -1).astype(np.float32), k=5)

# Persist: two files per index
faiss.write_index(index, "index.faiss")
# Companion JSON stores chunk metadata keyed by FAISS integer ID
```

Persistence: each index serialises to `{experiment_id}.faiss` + `{experiment_id}_chunks.json`. Loading is `faiss.read_index(path)` — no server, no daemon, no port conflicts.

## Alternatives Considered

| Option | Pros | Cons | Why Not |
|--------|------|------|---------|
| **FAISS** ✅ | Zero-dependency server, exact control over index type/dimension, per-file persistence, 10M+ vector scale | No built-in metadata storage, manual ID→chunk mapping needed | — (selected) |
| **ChromaDB** | Metadata built-in, Python-native, good for production APIs | Module-level EF singleton conflicts between configs, abstracts embedding injection, wrong granularity for experiment sweeps | P4 lesson: global singleton breaks multi-config tests |
| **Qdrant** | Rich filtering, production-ready | Requires running server process, adds ops overhead | Experiment platform not a production service |
| **numpy + cosine_similarity** | Zero deps | O(n) scan, no index, 35 configs × 4 papers × 500 chunks = slow | Performance unacceptable beyond toy scale |

---

## Quantified Validation

- P4 ChromaDB bug took 2 hours to diagnose: `normalize_embeddings()` return type changed silently.
- FAISS `IndexFlatIP` search on 2000 vectors (4 papers × ~500 chunks): < 1ms per query — fast enough that latency differences across 35 configs are measurement noise, not index overhead.
- Persistence: `.faiss` binary write + JSON metadata write for 2000 vectors ≈ 12ms total.

---

## Consequences

**Easier:** Each experiment config is fully isolated (separate files). Index dimension is explicit at construction — dimension mismatch is a hard error, not a silent failure. No server process to manage.

**Harder:** Manual ID→chunk mapping required (integer FAISS IDs must map back to `Chunk` objects). Metadata filtering (e.g., "only return chunks from page 3") requires post-retrieval filtering rather than native index queries.

**Portability:** P6–P9 can import the `FAISSVectorStore` class directly if they need a local vector index. The `BaseVectorStore` ABC ensures any future swap (Qdrant for production) only touches the implementation class.

## Java/TS Parallel

FAISS is analogous to a hand-rolled in-memory `Map<Integer, float[]>` with a binary search index, while ChromaDB is like Spring Data JPA with annotations — it does more, but you control less. For an experiment platform where you need to swap the underlying store between runs, the lower-level primitive is the right choice.

```java
// ChromaDB equivalent (Spring Data JPA style) — hides index internals
@Repository
class EmbeddingRepository extends JpaRepository<Chunk, UUID> {}

// FAISS equivalent (explicit control) — you own the index lifecycle
IndexFlatIP index = new IndexFlatIP(384);
index.add(vectors);
float[][] results = index.search(query, topK);
```

## Interview Signal

Choosing FAISS over ChromaDB demonstrates tool-selection maturity: the right tool depends on the job. ChromaDB optimises for production API ergonomics; FAISS optimises for programmatic experiment control. Using a production-oriented tool for an experiment platform creates accidental complexity (global state, server processes, opaque abstractions) that would slow down iteration. Recognising this trade-off — and documenting the P4 lesson that informed it — signals systematic engineering judgment over cargo-culting popular tools.
