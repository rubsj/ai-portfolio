# ADR-002: No LangChain — First-Principles RAG

**Project:** P5: ShopTalk Knowledge Agent
**Category:** Architecture
**Status:** Accepted
**Date:** 2026-03-03

---

## Context

LangChain provides `RecursiveCharacterTextSplitter`, `RetrievalQA`, `HuggingFaceEmbeddings`, and dozens of pre-built chains for RAG pipelines. The obvious choice for a RAG project would be to use LangChain.

P5's purpose is to demonstrate that we understand *why* RAG works, not just *how to configure* a LangChain chain. Specifically:
- The portfolio must show depth of understanding for EM-level interviews at Tier-1 companies.
- P5 is an experiment platform comparing chunking strategies and retriever types — the experiment variables *are* the implementation details LangChain would hide.
- A portfolio that says "I configured LangChain" signals configuration skill; a portfolio that says "I built the chunking pipeline, designed the retriever abstractions, and measured the performance tradeoffs" signals engineering judgment.

---

## Decision

Build all RAG components from first principles behind Abstract Base Classes (ABCs), with no LangChain dependency.

Six interfaces, each with a Java-parallel ABC pattern:

```python
# Python ABCs (src/interfaces.py)
class BaseChunker(ABC):
    @abstractmethod
    def chunk(self, document: Document) -> list[Chunk]: ...

class BaseEmbedder(ABC):
    @abstractmethod
    def embed(self, texts: list[str]) -> np.ndarray: ...
    @property
    @abstractmethod
    def dimensions(self) -> int: ...

class BaseVectorStore(ABC):
    @abstractmethod
    def add(self, chunks: list[Chunk], embeddings: np.ndarray) -> None: ...
    @abstractmethod
    def search(self, query_embedding: np.ndarray, top_k: int) -> list[tuple[Chunk, float]]: ...
```

Five chunking strategy implementations (all extend `BaseChunker`):
- `FixedSizeChunker` — character sliding window baseline
- `RecursiveChunker` — separator hierarchy `["\n\n", "\n", ". ", " ", ""]`
- `SlidingWindowChunker` — tiktoken-based token windows
- `HeadingSemanticChunker` — regex section boundary detection
- `EmbeddingSemanticChunker` — cosine similarity breakpoints via MiniLM

Each chunker is independently testable (53 unit tests, no PDFs required).

## Alternatives Considered

| Option | Pros | Cons | Why Not |
|--------|------|------|---------|
| **First-principles ABCs** ✅ | Full control, testable in isolation, demonstrates understanding, enables precise experiment measurement | More code to write | — (selected) |
| **LangChain** | Pre-built components, large community, fast to scaffold | Hides internals (can't inspect why a boundary was placed), version churn, difficult to unit-test without mocking LangChain internals | Portfolio signal would be "I used LangChain", not "I built RAG" |
| **Haystack** | RAG-focused, pipeline abstractions | Same problem as LangChain — abstracts the experiment variables away | — |
| **LlamaIndex** | Good for document indexing | Abstracts chunking and retrieval into nodes/pipelines | Same problem |

---

## Quantified Validation

- 5 chunkers implemented: Fixed, Recursive, Sliding Window, Heading-Semantic, Embedding-Semantic.
- 53 tests in `test_chunkers.py`: 53/53 passing, all without loading real PDF files or real models.
- `EmbeddingSemanticChunker` tested with mocked `SentenceTransformer` — the mock crafts orthogonal embeddings to place boundaries exactly where the test needs them. This level of unit-test precision is only possible when you own the implementation.
- `RecursiveChunker._split_text()` is independently testable: pass any text + separator list, get back segments. LangChain's equivalent is a class method that depends on the full splitter object.

---

## Consequences

**Easier:** Every component is independently benchmarkable. Swapping a chunker in an experiment config is one line (the `ExperimentConfig.chunking_strategy` field). Coverage is high because there are no third-party abstractions to stub out.

**Harder:** More boilerplate per component. Must implement retry/error handling without LangChain's `Runnable` infrastructure.

**Portability:** The `BaseChunker` / `BaseEmbedder` / `BaseRetriever` ABC pattern is reusable across P6–P9 wherever RAG components are needed. Any future migration to LangChain for a production project is a clean adapter implementation rather than a total rewrite.

## Java/TS Parallel

The ABC pattern maps directly to Java interfaces:

```java
// Java interface
public interface Chunker {
    List<Chunk> chunk(Document document);
}

public class FixedSizeChunker implements Chunker {
    @Override
    public List<Chunk> chunk(Document document) { ... }
}
```

```python
# Python ABC equivalent
class BaseChunker(ABC):
    @abstractmethod
    def chunk(self, document: Document) -> list[Chunk]: ...

class FixedSizeChunker(BaseChunker):
    def chunk(self, document: Document) -> list[Chunk]: ...
```

Using LangChain would be like depending on Spring's `TextSplitter` bean instead of defining your own `Splitter` interface — you lose the ability to mock, benchmark, and swap implementations independently.

---

## Interview Signal

"I can use LangChain — I chose not to for this project to prove I understand the internals." This decision demonstrates three things:
1. **Build-vs-buy reasoning**: knows when abstraction helps (production speed) vs. when it hurts (experiment precision, portfolio signal).
2. **Testability-first design**: ABC + dependency injection enables 100% unit-testable components without running PDFs or models.
3. **Measurement discipline**: you can only measure what you control; LangChain's internals would introduce unmeasurable variance into the chunking comparison.
